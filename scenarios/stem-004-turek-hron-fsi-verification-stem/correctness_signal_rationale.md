| Mesh motion is solved relative to the pristine reference configuration each step, driven by the *total* (not incremental) interface displacement | FSI Verification Protocol v1 §3 ("keep the mesh valid... for the full duration of the production case"); Numerical Methods Handbook §4 (moving-domain problems, "verify it the same way section 6 requires... not against 'it ran'") | Solving incrementally on the already-deformed mesh from the previous step compounds distortion independent of any stiffening parameter — measured directly: a compounding implementation failed at roughly half the displacement a reference-based implementation handled cleanly. The failure mode is silent mesh degradation (shrinking/inverting cells), not an immediate crash, so it can pass a short smoke run while being unsound for the production case |
| The nonlinear (solid or fluid) solve's convergence is cross-checked against an independent bound, not trusted on its own "converged" report | Numerical Methods Handbook §6 ("verification means showing our solver reproduces an independent result, not that it runs"); FSI Verification Protocol v1 §6 ("a real, reproducible comparison... not required to hit tight tolerance, but real") | Measured directly: a badly-configured nonlinear solve converged smoothly (SNES reported success) while producing a static deflection ~18x smaller than a from-scratch linear-elasticity cross-check, and mesh refinement made the discrepancy *worse* rather than better — the refinement direction is what distinguishes a real discretization issue from a solver-configuration bug, so a submission that doesn't attempt any such cross-check has no way to catch this class of error, and the "converged" flag alone gives false confidence |
| The production run is carried far enough in simulated time to observe whether the baffle's motion is settling to a fixed position or continuing to move, and that observation — not an assumption — determines what's reported | FSI Verification Protocol v1 §5 ("Characterise the tip displacement's behaviour... is it approaching a fixed value, or does it continue moving... Report which, and the basis for that conclusion"); Jira HLX-401 comment (Elena Voss, vendor bench-test observation: "settles into a large, sustained back-and-forth motion... does not just bend over and hold position") | A run stopped as soon as the baffle first deflects cannot distinguish "settles" from "oscillates" — reporting a static-looking deflection as the verified result when the case's defining behavior is sustained motion (per the bench-test context) mischaracterizes what was actually verified, the same category of error `stem-003`'s smoke-case trap was (reporting a number from before the physics of interest has developed) |
| Fluid-solid interface data (traction on the solid, velocity/displacement on the fluid) is applied with Newton's-third-law-consistent sign/direction at the shared boundary | FSI Verification Protocol v1 §2 (interface geometry pinned); Numerical Methods Handbook §5 (distributed-assembly discipline extended to "the solid and mesh-motion solves too, not only the fluid") | A sign error at the interface often still produces a run that completes and looks superficially plausible (some deflection, some flow) while being quantitatively and sometimes qualitatively wrong (e.g. deflection in the wrong direction, or a coupling that reads far stiffer/softer than the pinned material parameters imply) — exactly the "smooth, physical-looking curve that is simply wrong" failure shape `stem-003`'s coefficient-sign traps were |
| The coupling scheme's stability is actually checked over the required run duration, not assumed from a short smoke test | FSI Verification Protocol v1 §3 ("the study lead has not mandated a specific... coupling scheme... Whatever is chosen must... remain numerically stable for that same duration — if an approach becomes unstable partway through, that is a defect in the approach, not a property of the case to route around by shortening the run") | Measured directly, not hypothetical: a one-pass (non-iterated) staggered coupling implementation was stable and smooth for the first ~95% of the interval it was eventually run over, then showed increasingly noisy growth, then outright solver failure — the added-mass instability the FSI literature documents for this simplification at density ratios like this baffle's. A submission that only ran the short smoke case would not have seen this; the production-duration run is what actually exercises it |
| The reference comparison is a real, reproducible computation against `data/reference/ref_fsi2.point`, with column identities confirmed (not assumed) before being used | FSI Verification Protocol v1 §6 ("a real, reproducible comparison... not hardcoding or asserting reference values"); `data/README.md` ("column-by-column identification beyond time/timestep is still being confirmed — do not assume... without checking") | The reference file's column layout is genuinely not self-documenting (12 numeric columns, no header) — assuming a column mapping without verifying it against the reference's own definitions or cross-checked summary statistics risks comparing against the wrong quantity entirely while reporting a clean-looking "validated" result |

## Note on what is *not* in this table

Per `authentic-difficulty-principle.md`, the **choice** of coupling scheme (monolithic vs.
partitioned) and mesh-motion technique are explicitly *not* correctness signals by themselves —
only whether the chosen approach is actually stable and valid over the required duration is.
A submission that picks partitioned coupling and handles the stability question (sub-iteration,
relaxation, or an explicit justification) should score the same as one that picks monolithic and
handles it differently. See `correctness_signals.md`'s "Don't penalize for" section.
